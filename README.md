# Redrob Hackathon — Candidate Ranker

**Track:** The Data & AI Challenge — Intelligent Candidate Discovery & Ranking  
**JD target:** Senior AI Engineer — Founding Team @ Redrob AI

## How to run

```bash
# No external dependencies — uses Python stdlib only
python rank.py --candidates data/candidates.jsonl --out output/submission.csv
```

Runtime: **~30 seconds** for the full 100,000-candidate pool on a CPU-only machine.  
No GPU, no network calls, no hosted LLM API calls during ranking.

To view results in browser after running:
```bash
python launch_dashboard.py
```

For a different job description:
```bash
python rank.py --candidates data/candidates.jsonl --out output/submission.csv --jd data/job_description.txt
```

## Approach

The JD explicitly states that keyword-counting is a trap built into the dataset. So this ranker is structured around **career-history fit**, not skill bag size.

### Scoring pipeline (per candidate)

1. **Career-fit score (62% weight)** — reads `current_title`, all historical titles, and all description text. Strong ML/AI/search/ranking titles score highest. Off-target titles (Marketing Manager, HR Manager, etc.) get their score crushed to ~15% regardless of skills listed — directly countering the keyword-stuffer trap.

2. **Skill trust score (38% weight)** — matches skills against JD must-have categories, weighted by a trust factor from `endorsements` and `duration_months`. A skill with 0 endorsements and 0 months barely counts.

3. **Behavioral availability multiplier** — applied multiplicatively. Factors: `last_active_date`, `recruiter_response_rate`, `open_to_work_flag`, `notice_period_days`, `interview_completion_rate`.

4. **Location modifier** — Pune/Noida preferred (+8%); Tier-1 India cities welcome; outside India discounted (no visa sponsorship).

5. **Experience-band modifier** — soft band around 5–9 years, not a hard cutoff.

6. **Honeypot detection** — flags impossible profiles (expert proficiency with 0 duration, endorsements wildly disproportionate to tenure). Score multiplied by 0.02.

### Why not LLM-per-candidate

No GPU, no network, must finish in 5 minutes for 100K candidates. Pure Python + regex + arithmetic — fast, transparent, reproducible.

## Known limitations / honest caveats

- Title/skill matching is keyword + category-based, not a learned model — it
  cannot catch every paraphrase of "did production retrieval work." Some
  plain-language strong candidates whose descriptions use unusual phrasing
  may be under-scored.
- The honeypot heuristics are intentionally conservative (low false-positive
  bias) to avoid wrongly crushing legitimate non-trendy candidates; this
  means a small number of honeypots may not be caught.
- Weights (0.62 career / 0.38 skill, multiplier magnitudes, etc.) were set
  by reasoning from the JD text rather than fit against the hidden ground
  truth (which isn't available during the competition) — these are
  judgment calls and a clear area to iterate on if you get feedback.
