# Redrob Hackathon — Candidate Ranker

**Track:** The Data & AI Challenge — Intelligent Candidate Discovery & Ranking
**JD target:** Senior AI Engineer — Founding Team @ Redrob AI

## How to run

```bash
pip install -r requirements.txt   # stdlib only — this is a no-op, kept for spec compliance
python rank.py --candidates data/candidates.jsonl --out output/submission.csv
```

Runtime: **~30 seconds** for the full 100,000-candidate pool on a CPU-only machine.
No GPU, no network calls, no hosted LLM API calls during ranking — fully compliant
with the 5-minute / 16GB / CPU-only / no-network compute constraint in Section 3
of the submission spec.

If you only have `candidates.jsonl.gz`:

```bash
python rank.py --candidates data/candidates.jsonl.gz --out output/submission.csv
```
(gzip is handled automatically based on the file extension.)

## Approach

The JD explicitly states that keyword-counting is a trap built into the dataset:
*"A candidate who has all the AI keywords listed as skills but whose title is
'Marketing Manager' is not a fit, no matter how perfect their skill list looks."*
So this ranker is structured around **career-history fit**, not skill bag size.

### Scoring pipeline (per candidate)

1. **Career-fit score (62% weight)** — reads `current_title`, all historical
   `title`s, and all `description` text. Strong direct ML/AI/search/ranking
   titles score highest. Adjacent titles (backend/data engineer) only get
   credit if the role descriptions actually mention retrieval/ranking/NLP
   work. Off-target titles (Marketing Manager, HR Manager, etc.) get their
   score crushed to ~15%, regardless of skills listed — this directly counters
   the keyword-stuffer trap. Additional penalties apply for: research-only
   backgrounds with no production deployment evidence, vision/speech/robotics-
   only specialists with no NLP/IR exposure, careers spent entirely at pure
   consulting/services firms (TCS/Infosys/Wipro/etc.), and title-chasing
   patterns (3+ jobs, each ≤18 months).

2. **Skill trust score (38% weight)** — matches skills against the JD's four
   must-have categories (embeddings/retrieval, vector DB/hybrid search,
   Python, eval frameworks) and nice-to-haves (LoRA, learning-to-rank,
   HR-tech, etc.). Each skill's contribution is weighted by a *trust factor*
   derived from `endorsements` and `duration_months` — a skill listed with
   0 endorsements and 0 months behind it counts far less than one with real
   history, which directly discourages keyword-stuffing in the skills array.

3. **Behavioral availability multiplier** — applied multiplicatively on top
   of the base fit score, per the JD's explicit instruction: *"a perfect-on-
   paper candidate who hasn't logged in for 6 months and has a 5% recruiter
   response rate is, for hiring purposes, not actually available. Down-weight
   them appropriately."* Factors: `last_active_date` recency,
   `recruiter_response_rate`, `open_to_work_flag`, `notice_period_days`,
   `interview_completion_rate`.

4. **Location modifier** — Pune/Noida preferred (+8%); other Tier-1 Indian
   cities welcome (neutral); other India cities discounted unless
   `willing_to_relocate`; outside India discounted further (JD: no visa
   sponsorship, case-by-case).

5. **Experience-band modifier** — soft band around the JD's 5–9 year range
   rather than a hard cutoff, matching the JD's own framing ("a range, not a
   requirement... we'll seriously consider candidates outside the band if
   other signals are strong").

6. **Honeypot detection (near-hard filter)** — flags internally-impossible
   profiles: "expert" proficiency with ~0 `duration_months`, more expert
   skills than plausible given total experience, career-history duration
   that doesn't match `years_of_experience` or doesn't match its own
   start/end dates, and endorsement counts wildly disproportionate to
   duration. Flagged profiles have their score multiplied by 0.02 (pushed
   to the bottom, not zeroed, to keep sort ordering well-defined).

### Why this isn't an LLM-per-candidate system

The submission spec is explicit that hosted LLM calls per candidate won't fit
the 5-minute budget at real-world scale (200K candidates), and the ranking
step must run with no network access. Everything here is regex/keyword
matching plus arithmetic over structured JSON fields — no embeddings model,
no LLM calls, no GPU. This keeps the approach honest about the latency-quality
tradeoff the JD asks engineers to reason about explicitly.

## Known limitations / honest caveats

- Title/skill matching is keyword + category-based, not a learned model — it
  cannot catch every paraphrase of "did production retrieval work." Some
  plain-language strong candidates whose descriptions use unusual phrasing
  may be under-scored.
- The honeypot heuristics are intentionally conservative (low false-positive
  bias) to avoid wrongly crushing legitimate non-trendy candidates; this
  means a small number of honeypots may not be caught.
- Weights (0.62 career / 0.38 skill, multiplier magnitudes, etc.) were set
  by reasoning from the JD text rather than fit against the hidden ground
  truth (which isn't available during the competition) — these are
  judgment calls and a clear area to iterate on if you get feedback.
